# 小米超级岛服务进度管家 — 产品需求文档 (PRD)

## 1. 产品概述

### 1.1 核心定位

「小米超级岛服务进度管家」是一款飞书群聊机器人，用于管理和查询各应用接入**超级岛**、**负一屏小部件**的开发进度。它以飞书多维表格 (Bitable) 为底层数据库，通过自然语言交互、文件解析、卡片模板等多种方式实现进度数据的录入、更新与查询，让团队成员随时掌握全局进展。

### 1.2 目标用户

- 超级岛项目的产品经理、项目经理
- 各应用对接开发人员
- 岛运营、岛产品等相关干系人

### 1.3 使用场景

- 在飞书群聊中，通过发送文字、文件或指令，快速录入和查询应用接入进度。
- 需要汇总全局进度时，一键获取实时在线表格链接。

---

## 2. 数据模型

机器人管理的核心数据字段如下：

| 字段名 | 类型 | 说明 | 示例 |
|---|---|---|---|
| 应用名称 | 文本 | 接入应用的名称 | 拼多多、抖音 |
| 场景 | 单选 | 接入的业务场景 | 超级岛 / 负一屏小部件 |
| 大岛 | 文本 | 所属大岛分类 | 出行、购物 |
| 小岛 | 文本 | 所属小岛分类 | 打车、外卖 |
| 接入版本 | 文本 | 计划接入的系统版本号 | 26.4 |
| 进度排期 | 文本 | 当前进度或预计完成时间 | 开发中 / 2026-04-15 |
| 支持小窗 | 单选 | 是否支持小窗模式 | 是 / 否 / 待定 |
| 支持分享 | 单选 | 是否支持分享功能 | 是 / 否 / 待定 |

## 3. 功能详细设计

### 功能一：NLP 智能单条录入与更新

**触发方式**：用户在群聊中直接发送一段自然语言文本。

**交互流程**：

1. 用户发送文字消息，例如：`拼多多预计26.4接入超级岛，大岛购物，支持小窗`
2. 机器人调用 LLM 进行意图识别，判定为"录入/更新"意图。
3. LLM 从文本中提取结构化字段：
   - 应用名称：拼多多
   - 接入版本：26.4
   - 场景：超级岛
   - 大岛：购物
   - 支持小窗：是
4. 机器人在 Bitable 中查询是否已存在该应用记录：
   - 若存在 → 更新对应字段。
   - 若不存在 → 新增一条记录。
5. 机器人回复确认卡片，展示已录入/更新的字段摘要。

**边界处理**：

- 若 LLM 无法识别出有效字段，回复提示："抱歉，未能识别有效信息，请尝试更明确的描述或使用录入模板。"
- 若仅提取到部分字段（如只有应用名称），仍正常录入，缺失字段留空。

### 功能二：表格批量智能录入（核心增强）

#### 2a. 本地表格上传

**触发方式**：用户在群聊中直接发送 Excel (.xlsx) 或 CSV (.csv) 文件。

**交互流程**：

1. 用户上传文件到群聊。
2. 机器人检测到文件消息，判断文件类型为 Excel/CSV。
3. 通过飞书 API 下载文件到临时目录。
4. 使用 pandas 解析文件内容，进行列名智能映射（模糊匹配到标准字段）。
5. 逐行校验数据，跳过无效行，记录错误信息。
6. 将有效数据批量写入 Bitable。
7. 回复汇总卡片：成功 N 条，失败 M 条（附失败原因）。

**列名映射规则**：

| 用户可能的列名 | 映射到标准字段 |
|---|---|
| app / 应用 / 应用名 / 名称 | 应用名称 |
| version / 版本 / 接入版本 | 接入版本 |
| scene / 场景 | 场景 |
| 大岛 / 一级分类 | 大岛 |
| 小岛 / 二级分类 | 小岛 |
| 进度 / 排期 / 时间 | 进度排期 |
| 小窗 / 支持小窗 | 支持小窗 |
| 分享 / 支持分享 | 支持分享 |

#### 2b. 飞书表格链接解析

**触发方式**：用户在群聊中发送包含飞书多维表格 (Bitable) 或电子表格 (Sheets) 链接的消息。

**交互流程**：

1. 用户发送消息，例如：`请同步这个表格 https://xxx.feishu.cn/base/xxxxx`
2. 机器人通过正则匹配识别出飞书表格链接。
3. 调用飞书 API 读取源表格数据。
4. 进行字段映射和数据校验（逻辑同 2a）。
5. 将有效数据批量写入目标 Bitable。
6. 回复汇总卡片：成功 N 条，失败 M 条。

**支持的链接格式**：

- 飞书多维表格：`https://*.feishu.cn/base/*`
- 飞书电子表格：`https://*.feishu.cn/sheets/*`

### 功能三：卡片模板录入

**触发方式**：用户发送指令 `录入模板` 或 `模板`。

**交互流程**：

1. 用户发送 `录入模板`。
2. 机器人返回一张飞书互动卡片 (Message Card)，包含以下表单字段：
   - 应用名称（文本输入）
   - 场景（下拉选择：超级岛 / 负一屏小部件）
   - 大岛（文本输入）
   - 小岛（文本输入）
   - 接入版本（文本输入）
   - 进度排期（文本输入）
   - 支持小窗（下拉选择：是 / 否 / 待定）
   - 支持分享（下拉选择：是 / 否 / 待定）
   - 提交按钮
3. 用户填写并点击提交。
4. 机器人接收卡片回调，将数据写入 Bitable。
5. 回复确认消息："✅ 已成功录入 [应用名称] 的进度信息。"

---

### 功能四：进度查询与在线表格汇总（核心增强）

#### 4a. 单应用查询

**触发方式**：用户发送包含查询意图的文本，例如 `查一下拼多多的进度` 或 `拼多多进度`。

**交互流程**：

1. LLM 识别为"查询"意图，提取应用名称。
2. 机器人在 Bitable 中检索该应用的记录。
3. 若找到 → 返回详情卡片，展示所有字段。
4. 若未找到 → 回复："未找到 [应用名称] 的相关记录。"

#### 4b. 进度汇总

**触发方式**：用户发送 `进度汇总` 或 `汇总`。

**交互流程**：

1. 机器人从 Bitable 读取全部记录。
2. 生成概览统计：
   - 总应用数
   - 各场景分布（超级岛 N 个，负一屏小部件 M 个）
   - 各进度状态分布
3. 返回汇总卡片，包含：
   - 统计概览信息
   - **可直接点击打开的飞书多维表格链接**（该表格即为机器人的底层 Bitable，实时同步最新数据）
   - 链接文案示例："📊 点击查看完整进度表格"

### 功能六：录入权限控制

**机制**：基于飞书 open_id 白名单，区分"可录入用户"和"只读用户"。

- 白名单用户：可执行所有操作（录入、查询、上传文件、同步链接）。
- 非白名单用户：只能执行查询类操作（查进度、进度汇总、对接人），录入请求会收到友好提示。
- 白名单通过 `.env` 中的 `ADMIN_OPEN_IDS` 配置，逗号分隔。
- 若 `ADMIN_OPEN_IDS` 为空，则所有人都有录入权限（开发阶段默认行为）。

---

### 功能七：操作日志记录

**机制**：所有用户与机器人的交互自动记录到 Bitable「操作日志」表。

**记录字段**：

| 字段 | 说明 | 示例 |
|---|---|---|
| 时间 | 交互发生时间 | 2026-03-27 21:38:00 |
| 用户ID | 发送者的飞书 open_id | ou_3f99be... |
| 用户消息 | 用户发送的原始文本 | 京东外卖，预计4月17号接入 |
| 机器人回复 | 机器人的回复内容 | ✅ 已成功将京东的进度录入... |
| 操作类型 | 操作分类 | 录入成功 / 查询 / 未识别 |

**触发方式**：自动记录，无需用户操作。管理员可在 Bitable 中查看完整交互历史。

## 4. 指令与路由优先级

机器人收到消息后，按以下优先级进行路由分发：

| 优先级 | 匹配规则 | 处理方式 |
|---|---|---|
| 1 | 文件消息（Excel/CSV） | → 功能二a：本地表格解析 |
| 2 | 精确指令：`录入模板` / `模板` | → 功能三：返回卡片模板 |
| 3 | 精确指令：`对接人` | → 功能五：返回对接人卡片 |
| 4 | 精确指令：`进度汇总` / `汇总` | → 功能四b：返回汇总卡片 |
| 5 | 文本中包含飞书表格链接 | → 功能二b：飞书表格解析 |
| 6 | 其他文本消息 | → LLM 意图识别（录入/更新/查询） |

---

## 5. 非功能性需求

- **响应时间**：普通文本交互 < 3 秒；文件解析 < 10 秒（100 行以内）。
- **错误处理**：所有异常场景需有友好的中文提示，不暴露技术细节。
- **数据安全**：临时下载的文件在处理完成后立即删除。
- **幂等性**：同一应用名称的重复录入视为更新，不产生重复记录。
- **高可用**：LLM 采用双擎架构（主力线上模型 + 本地兜底模型），主力模型超时或异常时自动降级到本地模型，确保服务不中断。
- **合并单元格处理**：Excel 文件中的合并单元格（如应用名称列）自动向下填充，确保子行数据完整。
